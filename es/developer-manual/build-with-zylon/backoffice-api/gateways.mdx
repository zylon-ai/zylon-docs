Un **Gateway** es un límite de ejecución aislado para tráfico API en Zylon. No es un dispositivo físico.

Los gateways se usan para:

- aislar entornos API por equipo o caso de uso
- controlar quién puede emitir y usar tokens en cada entorno
- acotar acceso a datos y actividad mediante políticas por gateway

## Requisitos previos

- Token con permisos de backoffice en `Authorization: Bearer {API_TOKEN}`
- URL base objetivo (`https://{BASE_URL}`)

## Gateways

### Crear un gateway

```bash
curl -X POST "https://{BASE_URL}/api/v1/platform/gateway" \
  -H "Authorization: Bearer {API_TOKEN}" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "docs-gw-final",
    "description": "Docs final examples",
    "interceptors": []
  }'
```

## Tokens de gateway

Los tokens de gateway están pensados para acceso a Workspace API.

### Crear un token de gateway

```bash
curl -X POST "https://{BASE_URL}/api/v1/platform/gateway/{gatewayId}/token" \
  -H "Authorization: Bearer {API_TOKEN}" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "docs-gw-token-final",
    "scopes": ["Workspace.ReadWrite.All"],
    "valid_until": "2027-12-31T00:00:00Z",
    "revoke_existing": false
  }'
```

<Accordion title="Ejemplo de respuesta en demo (400)">
```json
{
  "type": "BadRequest",
  "status": 400,
  "message": "At least one valid scope must be specified for the API token"
}
```
</Accordion>

### Listar tokens de gateway

```bash
curl "https://{BASE_URL}/api/v1/platform/gateway/{gatewayId}/token?page_size=2" \
  -H "Authorization: Bearer {API_TOKEN}"
```

<Accordion title="Ejemplo de respuesta">
```json
{
  "data": [],
  "has_next_page": false,
  "has_previous_page": false
}
```
</Accordion>

### Actualizar token de gateway

```bash
curl -X PUT "https://{BASE_URL}/api/v1/platform/gateway/{gatewayId}/token/{token_hash}" \
  -H "Authorization: Bearer {API_TOKEN}" \
  -H "Content-Type: application/json" \
  -d '{"name": "docs-gw-token-final-rotated"}'
```

<Accordion title="Ejemplo de respuesta">
```text
Actualmente este endpoint no documenta un cuerpo de respuesta.
Usa HTTP 2xx como señal de éxito.
```
</Accordion>

### Eliminar token de gateway

```bash
curl -X DELETE "https://{BASE_URL}/api/v1/platform/gateway/{gatewayId}/token/{token_hash}" \
  -H "Authorization: Bearer {API_TOKEN}"
```

<Accordion title="Ejemplo de respuesta (204)">
```text
Sin cuerpo de respuesta.
```
</Accordion>

## Errores comunes

- Omitir `interceptors` en payloads de creación/actualización de gateway.
- Asumir que cualquier string de scope es válido al crear tokens de gateway.
- Mezclar account IDs y user IDs en operaciones de usuarios de gateway.

## Errores y casos límite

- **400**: body inválido o scope de token no soportado.
- **404**: gateway o usuario de gateway no encontrado.
- **204**: eliminaciones exitosas de token o gateway pueden no devolver body.
