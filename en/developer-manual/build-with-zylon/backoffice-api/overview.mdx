The Backoffice API is used for platform administration tasks: seat usage, gateway lifecycle, gateway users, and gateway tokens.

## Base URL

All Backoffice endpoints live under:

```bash
https://{BASE_URL}/api/v1
```

## Authentication

Backoffice requests require an API token with admin-level permissions.

```bash
-H "Authorization: Bearer {API_TOKEN}"
```

Unlike Workspace `/api/v1/app/*` endpoints, Backoffice endpoints do not require the `x-org` header.

## Core domains

| Domain | Purpose |
| --- | --- |
| License usage | Track seat limits and real-time seat consumption. |
| Accounts | Manage account CRUD and account tokens for backoffice operations. |
| Gateways | Create and manage gateways, gateway users, and gateway-scoped tokens. |

## Check your instance limitations

Use `GET /api/v1/license/usage` to check seat usage:

```bash
curl "https://{BASE_URL}/api/v1/license/usage" \
  -H "Authorization: Bearer {API_TOKEN}"
```

<Accordion title="Example response">
```json
{
  "workspace_seats": 10000,
  "used_workspace_seats": 141,
  "operator_seats": 30,
  "used_operator_seats": 2,
  "developer_seats": 100,
  "used_developer_seats": 7
}
```
</Accordion>

## Common mistakes

- Using a workspace token with insufficient backoffice permissions.
- Sending `x-org` and assuming it changes backoffice behavior.
- Treating seat usage counters as writable values.

## Errors and edge cases

- **401/403**: missing token or missing backoffice permissions.
- **404**: referenced gateway/user/token does not exist.
- Some legacy operations expose minimal error payloads; rely on HTTP status and endpoint context.
