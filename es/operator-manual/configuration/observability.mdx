Zylon incluye dos herramientas de observabilidad habilitadas por defecto:

- **Sentry** para informes de fallos: ayuda a diagnosticar malfuncionamientos de la plataforma
- **Grafana** para métricas de uso anónimas y agregadas: ayuda a entender cómo se utiliza Zylon para agilizar el soporte

Ninguna de las dos herramientas envía información privada.

## Dominios de red a incluir en la lista blanca

Si tu clúster restringe el tráfico saliente, los siguientes dominios deben añadirse a la lista blanca para la telemetría de uso:

- `*.grafana.net`

Para mayor detalle, los endpoints específicos utilizados son:

- Métricas de uso básico: [https://prometheus-prod-24-prod-eu-west-2.grafana.net](https://prometheus-prod-24-prod-eu-west-2.grafana.net/)
- Trazas del sistema: [https://tempo-prod-10-prod-eu-west-2.grafana.net](https://tempo-prod-10-prod-eu-west-2.grafana.net/)

## Configuración de observabilidad

Todos los ajustes de observabilidad se encuentran bajo la clave `observability` en el fichero de configuratión. A continuación se muestra la referencia completa con todas las claves disponibles.

```yaml
observability:
  crashReporting: true          
  monitoring: true              
  usageMetrics: true            
  platformMetrics:
    enabled: false              
    generationIntervalMs: 2000  
    gpu:
      enabled: true             
    inference:
      counterLatencies: true   
      histogramLatencies: true 
      summaryLatencies: true   
      summaryQuantiles: ""     
```

### Referencia de indicadores

| Indicador | Valor por defecto | Descripción |
|---|---|---|
| `crashReporting` | `true` | Envía informes de fallos a Sentry. Desactívalo si no deseas telemetría externa. |
| `monitoring` | `true` | Interruptor principal. Despliega la infraestructura de monitorización (Prometheus, Grafana, k8s-monitoring). Debe estar en `true` para que se recopile cualquier métrica. |
| `usageMetrics` | `true` | Envía telemetría de uso anónima y agregada a Grafana Cloud. No se incluyen datos privados. |
| `platformMetrics.enabled` | `false` | Habilita la recopilación de métricas Prometheus de Triton y vLLM. Requiere `monitoring: true`. |
| `platformMetrics.generationIntervalMs` | `2000` | Intervalo en milisegundos con el que Triton genera métricas internamente. No es el intervalo de recopilación. |
| `platformMetrics.gpu.enabled` | `true` | Incluye métricas de GPU junto con las métricas de inferencia. |
| `platformMetrics.inference.counterLatencies` | `true` | Expone contadores de latencia acumulativos (`nv_inference_*_duration_us`). |
| `platformMetrics.inference.histogramLatencies` | `true` | Expone histogramas de latencia (`nv_inference_first_response_histogram_ms`). |
| `platformMetrics.inference.summaryLatencies` | `true` | Expone cuantiles de resumen con ventana deslizante (`nv_inference_*_summary_us`). |
| `platformMetrics.inference.summaryQuantiles` | `""` | Sobreescribe los cuantiles por defecto. Formato: `"<cuantil>:<error>,..."`. Déjalo vacío para usar los valores predeterminados de Triton. |

### Desactivar la observabilidad por completo

Para desactivar toda la monitorización y telemetría:

```yaml
observability:
  crashReporting: false
  monitoring: false
  usageMetrics: false
```

### Habilitar la recopilación de métricas de plataforma

Desplegar la infraestructura de monitorización no inicia la recopilación por sí sola. La recopilación solo comienza cuando la clave de métricas específicos también están configurados en `true`:

```yaml
observability:
  monitoring: true
  platformMetrics:
    enabled: true
```

## Envío de métricas a un destino externo

Por defecto, el stack de monitorización recopila métricas internamente. Para reenviar métricas a un receptor externo, como tu propip stack de Grafana Cloud, un Prometheus auto-alojado u otro endpoint compatible, utiliza `k8s-monitoring.extraDestinations` en tu archivo de configuration.

Cada entrada bajo `extraDestinations` es un destino con nombre. Puedes añadir múltiples destinos añadiendo más entradas con nombre:

```yaml
k8s-monitoring:
  extraDestinations:
    my-grafana:
      type: prometheus
      url: https://prometheus-prod-58-prod-eu-central-0.grafana.net/api/prom/push
      auth:
        type: basic
        username: "<tu-usuario>"
        password: "<tu-api-token>"
    my-open-telemetry:
      type: otlp
      url: https://your-otel.com/push
      auth:
        type: basic
        username: "<usuario>"
        password: "<token>"
```

Los tipos de destino compatibles son `prometheus` y `otlp`.

## Panel de control de Grafana

Hay disponible un panel de control de Grafana de referencia para visualizar las métricas de Triton Inference Server y vLLM. Se ofrece como punto de partida que puedes importar, personalizar y ampliar según tus propias necesidades de observabilidad. La definición del panel se encuentra en [grafana-dashboard.json](https://raw.githubusercontent.com/zylon-ai/zylon-docs/main/snippets/grafana-dashboard.json).

Las métricas que aparecen en el panel provienen de dos fuentes, ambas recopiladas a través del endpoint `/metrics` del servicio Triton:

- **Triton Inference Server**: recuentos de peticiones, desgloses de latencia, estado de la GPU y estadísticas de cola. Consulta la [referencia de métricas de Triton](https://docs.nvidia.com/deeplearning/triton-inference-server/user-guide/docs/user_guide/metrics.html) para la lista completa.
- **vLLM**: estado del planificador, utilización de la caché KV, rendimiento de tokens e histogramas de latencia, reenviados por Triton a través de su API de métricas personalizadas. Consulta la [referencia de métricas de vLLM](https://docs.vllm.ai/en/stable/usage/metrics/) para la lista completa.

### Variables del panel de control

El panel de control expone cuatro filtros desplegables en la parte superior:

| Variable | Descripción |
|---|---|
| **Datasource** | Fuente de datos de Prometheus a consultar |
| **Environment** | Filtrar por etiqueta de entorno de despliegue (identificador de tu empresa) |
| **Model** | Filtrar por nombre del modelo servido por Triton |
| **GPU** | Filtrar por `gpu_uuid` para paneles específicos de GPU |

### Secciones del panel de control

El panel de control está organizado en torno a un flujo de investigación natural: comenzar con el resumen general de salud, profundizar en el rendimiento, luego en la latencia, entender *por qué* la latencia puede ser elevada mediante señales del planificador y capacidad, analizar la forma de la carga de trabajo y, finalmente, comprobar el estado del hardware.

Descarga el archivo [grafana-dashboard.json](https://raw.githubusercontent.com/zylon-ai/zylon-docs/main/snippets/grafana-dashboard.json) e impórtalo en Grafana mediante **Dashboards → Nuevo → Importar**. Para los pasos detallados, consulta la [documentación de importación de paneles de Grafana](https://grafana.com/docs/grafana/latest/visualizations/dashboards/build-dashboards/import-dashboards/).

| # | Sección | Pregunta que responde |
|---|---|---|
| 1 | Resumen (Overview) | ¿Cómo es la carga de trabajo a primera vista? |
| 2 | Rendimiento y errores (Throughput & Errors)| ¿Cuánto está gestionando? |
| 3 | Latencia (Latency)| ¿Qué tan rápido es? |
| 4 | Capacidad y planificador (Capacity & Scheduler) | ¿Por qué podría ser lento? |
| 5 | Análisis de carga de trabajo (Workload Analysis) | ¿Cómo es la carga? |
| 6 | Estado de la GPU | ¿Está la GPU bajo presión? |
| 7 | Recursos del host (Host Resources) | ¿Está el nodo en buen estado? |

Vista unificada de la latencia tanto en la capa de Triton como en la de vLLM. Primero aparecen los promedios a nivel de Triton y los desgloses por fase, seguidos de los percentiles de histograma de vLLM (TTFT, TPOT, E2E) y los tiempos de fase (prefill/decode).

![Cola media, cómputo y sobrecarga de E/S; percentiles TTFT de Triton y vLLM](/images/operator-manual/observability/latency-2.png)

![Tiempo por token de salida, latencia extremo a extremo, tiempo de prefill y decode, cuantiles de resumen de Triton](/images/operator-manual/observability/latency-3.png)

| Panel | Métrica(s) | Descripción |
|---|---|---|
| **Latencia promedio de extremo a extremo por modelo (ms)** | `nv_inference_request_duration_us`, `nv_inference_request_success` | Duración total promedio de la solicitud desde la entrada hasta la respuesta, incluyendo la espera en cola y el cómputo. |
| **Cascada de latencia (promedio apilado en ms)** | `nv_inference_queue_duration_us`, `nv_inference_compute_input_duration_us`, `nv_inference_compute_infer_duration_us`, `nv_inference_compute_output_duration_us` | Gráfico apilado que muestra el tiempo empleado en cada fase: espera en cola, preprocesamiento de entrada, cómputo del modelo y postprocesamiento de salida. Un segmento de cola dominante indica un problema de capacidad; un segmento de cómputo dominante significa que el modelo en sí es el cuello de botella. |
| **Tiempo promedio de espera en cola por modelo (ms)** | `nv_inference_queue_duration_us` | Tiempo que cada solicitud pasa esperando en la cola del planificador de Triton antes de que comience la ejecución. |
| **Tiempo promedio de cómputo del modelo por modelo (ms)** | `nv_inference_compute_infer_duration_us`, `nv_inference_exec_count` | Tiempo que el backend del modelo dedica a ejecutar la inferencia, normalizado por lote de ejecución. |
| **Sobrecarga promedio de E/S de cómputo por modelo (ms)** | `nv_inference_compute_input_duration_us`, `nv_inference_compute_output_duration_us` | Tiempo dedicado al preprocesamiento de entrada y postprocesamiento de salida, fuera de la ejecución real del modelo. |
| **Tiempo de Triton hasta la primera respuesta (P50 / P90 / P99, ms)** | `nv_inference_first_response_histogram_ms` | Latencia de extremo a extremo hasta el primer token de respuesta según lo observado por Triton. |
| **Tiempo de vLLM hasta el primer token - TTFT (P50 / P90 / P99)** | `vllm_llms_v1:time_to_first_token_seconds_bucket`, `vllm_embeddings_v1:time_to_first_token_seconds_bucket` | Percentiles de TTFT reportados por el backend de vLLM para modelos LLM y de embeddings. |
| **Tiempo por token de salida (P50 / P90 / P99)** | `vllm_llms_v1:time_per_output_token_seconds_bucket` | Latencia entre tokens (tiempo entre tokens generados consecutivamente). Solo es significativo para modelos LLM. |
| **Latencia de solicitud de extremo a extremo (P50 / P90 / P99)** | `vllm_llms_v1:e2e_request_latency_seconds_bucket`, `vllm_embeddings_v1:e2e_request_latency_seconds_bucket` | Latencia completa de la solicitud desde la perspectiva de vLLM, abarcando tanto modelos LLM como de embeddings. |
| **Tiempo de prefill y decode** | `vllm_llms_v1:request_prefill_time_seconds_sum`, `vllm_llms_v1:request_decode_time_seconds_sum`, `vllm_embeddings_v1:*` | Tiempo por fase dentro de vLLM: el prefill procesa el prompt completo, el decode genera cada token de salida de forma incremental. |
| **Percentiles de latencia de cola y cómputo de Triton (µs)** | `nv_inference_queue_summary_us`, `nv_inference_compute_infer_summary_us` | Cuantiles de ventana deslizante precalculados a partir de las métricas de resumen de Triton. Tiempo de espera en cola y tiempo de cómputo del modelo en P50/P99. |

### 4. Capacidad y planificador

Señales que explican *por qué* la latencia puede ser elevada. Si la latencia es alta, comprueba primero la presión sobre la caché KV y las interrupciones, luego la profundidad de la cola del planificador y el tamaño del lote de iteraciones.

![Paneles de capacidad y planificador](/images/operator-manual/observability/capacity-and-scheduler.png)

| Panel | Métrica(s) | Descripción |
|---|---|---|
| **Estado del planificador de vLLM (En ejecución / En espera)** | `vllm_llms_v1:num_requests_running`, `vllm_llms_v1:num_requests_waiting`, `vllm_embeddings_v1:num_requests_running`, `vllm_embeddings_v1:num_requests_waiting` | Número de peticiones en estado EN EJECUCIÓN y EN ESPERA dentro del planificador de vLLM a lo largo del tiempo. |
| **Utilización de caché KV de vLLM** | `vllm_llms_v1:kv_cache_usage_perc` | Porcentaje de bloques de caché KV actualmente en uso. Al acercarse al 100 % se activa la cola de peticiones o la interrupción. Umbrales: verde < 70 %, amarillo 70–89 %, rojo ≥ 90 %. |
| **Tiempo de espera en cola de peticiones (vLLM)** | `vllm_llms_v1:request_queue_time_seconds_sum`, `vllm_embeddings_v1:request_queue_time_seconds_sum` | Tiempo que cada solicitud pasó esperando en la cola interna de vLLM antes de la ejecución. |
| **Tasa de interrupciones** | `vllm_llms_v1:num_preemptions_total`, `vllm_embeddings_v1:num_preemptions_total` | Tasa de interrupciones del planificador. Cada interrupción desaloja una solicitud en ejecución de la caché KV debido a presión de memoria. Una tasa sostenida distinta de cero indica que la caché KV es demasiado pequeña para la carga actual. |
| **Tamaño del lote de iteración (P50 / P90 / P99, tokens)** | `vllm_llms_v1:iteration_tokens_total_bucket`, `vllm_embeddings_v1:iteration_tokens_total_bucket` | Distribución del total de tokens procesados por paso del motor (prompt + generación). Un P50 bajo en relación con el tamaño máximo de lote configurado indica que la GPU está infrautilizada; un P99 alto señala lotes ocasionalmente muy grandes. |
