La Backoffice API se usa para tareas de administración de plataforma: uso de licencias, ciclo de vida de gateways, usuarios de gateway y tokens de gateway.

## Base URL

Todos los endpoints de Backoffice viven bajo:

```bash
https://{BASE_URL}/api/v1
```

## Autenticación

Las solicitudes de Backoffice requieren un token con permisos de administración.

```bash
-H "Authorization: Bearer {API_TOKEN}"
```

A diferencia de los endpoints de Workspace `/api/v1/app/*`, los endpoints de Backoffice no requieren la cabecera `x-org`.

## Dominios clave

| Dominio | Propósito |
| --- | --- |
| Uso de licencias | Seguir límites de asientos y consumo real. |
| Cuentas | Gestionar CRUD de cuentas y tokens de cuenta para operaciones de backoffice. |
| Gateways | Crear y gestionar gateways, usuarios de gateway y tokens asociados. |

## Verifica las limitaciones de tu instancia

Usa `GET /api/v1/license/usage` para consultar el uso de licencias:

```bash
curl "https://{BASE_URL}/api/v1/license/usage" \
  -H "Authorization: Bearer {API_TOKEN}"
```

<Accordion title="Ejemplo de respuesta">
```json
{
  "workspace_seats": 100,
  "used_workspace_seats": 72,
  "operator_seats": 10,
  "used_operator_seats": 6,
  "developer_seats": 20,
  "used_developer_seats": 11
}
```
</Accordion>

## Errores comunes

- Usar un token de workspace sin permisos de backoffice.
- Enviar `x-org` asumiendo que altera el comportamiento de backoffice.
- Tratar los contadores de licencias como valores editables.

## Errores y casos límite

- **401/403**: token ausente o sin permisos de backoffice.
- **404**: gateway/usuario/token inexistente.
- Algunas operaciones heredadas devuelven payloads de error mínimos; usa el código HTTP y el contexto del endpoint.
