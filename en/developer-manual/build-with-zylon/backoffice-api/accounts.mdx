Use Backoffice account endpoints to manage platform identities and account-scoped API tokens.

## Prerequisites

- Backoffice-capable token in `Authorization: Bearer {API_TOKEN}`
- Target base URL (`https://{BASE_URL}`)

Before assigning or updating roles, review [Roles and permissions](/en/user-manual/backoffice/roles-and-permissions).

## Accounts

### Available account roles

Account roles control platform-level access:

- `Workspace`: Uses Workspace for projects and collaboration. No Platform Backoffice access.
- `Developer`: Works with gateways and API tokens for development. Access to Developer Tools.
- `Operator`: Manages accounts, roles, gateways, and platform operations. Full Platform Backoffice access.
- `SuperAdmin`: Initial administrative account for platform setup and governance. Same platform access as `Operator`.

### Create an account

Use `POST /api/v1/auth/register` to create a credentials-based account.

```bash
curl -X POST "https://{BASE_URL}/api/v1/auth/register" \
  -H "Authorization: Bearer {API_TOKEN}" \
  -H "Content-Type: application/json" \
  -d '{
    "email": "{username}",
    "password": "{password}",
    "roles": ["Workspace"]
  }'
```

### Update current account password

```bash
curl -X PUT "https://{BASE_URL}/api/v1/account/me" \
  -H "Authorization: Bearer {API_TOKEN}" \
  -H "Content-Type: application/json" \
  -d '{
    "current_password": "{current_password}",
    "new_password": "{new_password}"
  }'
```

<Accordion title="Example response (400 if current password is invalid)">
```json
{
  "type": "InvalidPassword",
  "status": 400,
  "message": "Invalid current password"
}
```
</Accordion>

## Tokens

Account tokens are intended for PrivateGPT API and Workspace API access.

- `gateway_id` is required when creating account tokens.
- To create tokens for Workspace-only usage, use [Gateways - Create a gateway token](/en/developer-manual/build-with-zylon/backoffice-api/gateways#create-a-gateway-token).

### Create account API token

```bash
curl -X POST "https://{BASE_URL}/api/v1/account/token" \
  -H "Authorization: Bearer {API_TOKEN}" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "docs-account-final-1772554276",
    "gateway_id": "{gatewayId}",
    "scopes": ["Workspace.ReadWrite.All"],
    "valid_until": "2027-12-31T00:00:00Z",
    "revoke_existing": false
  }'
```

```bash

  -H "Authorization: Bearer {API_TOKEN}"
```

## Common mistakes

- Mixing query-token and path-token patterns when working with `/api/v1/account/token/{token_hash}`.
- Updating account roles with workspace-only tokens that do not have backoffice permissions.
- Omitting `Content-Type: application/json` on body-based endpoints.

## Errors and edge cases

- **401/403**: missing token or insufficient backoffice permissions.
- **404**: account, organization, or token not found.
- Some update/delete operations may return empty bodies.
